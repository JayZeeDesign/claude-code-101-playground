# CLAUDE.md

This file contains instructions for Claude Code. Read all of it before doing anything.

## Project overview

Skateshop is an open source e-commerce storefront built with Next.js 14 App Router, React 18, TypeScript, Tailwind CSS, shadcn/ui, Drizzle ORM on Postgres, Clerk for auth, Stripe for payments, and uploadthing for file uploads. It uses pnpm as the package manager. Do not use npm or yarn. The app directory is src/app. Components live in src/components. Database code lives in src/db. Server actions live in src/lib/actions. Queries live in src/lib/queries. Zod validation schemas live in src/lib/validations. Always check these folders before creating new files.

## Frontend rules

- Always use functional components. Never use class components.
- Use the cn() helper from @/lib/utils for conditional classNames. Never concatenate class strings manually.
- All new UI components must use shadcn/ui primitives from src/components/ui. Do not install other component libraries.
- Use lucide-react icons via @radix-ui/react-icons. Actually we use @radix-ui/react-icons, check the existing imports first because some files use lucide.
- Pages in src/app must be server components by default. Only add "use client" when the component uses hooks or browser APIs.
- Use next/image for all images. Set width and height explicitly. Remote images must have their domains whitelisted in next.config.js.
- Forms must use react-hook-form with zodResolver. The schema goes in src/lib/validations, not inline.
- Loading states use the Skeleton component from src/components/ui/skeleton.tsx.
- Toasts use sonner. Import { toast } from "sonner". Do not use the old useToast hook, it was removed in the shadcn migration but some docs still reference it.
- Use the Shell component (src/components/shell.tsx) as the page wrapper for dashboard pages.
- Buttons that trigger server actions need a loading spinner. Copy the pattern from src/components/checkout/add-to-cart-form.tsx.
- Breakpoints: sm 640px, md 768px, lg 1024px, xl 1280px, 2xl 1400px (container). Mobile first, always.
- Fonts are Geist Sans and Geist Mono, loaded in src/lib/fonts.ts. Do not add Google Fonts.
- Dark mode uses next-themes. Test both themes for any UI change.
- Do not edit files in src/components/ui directly, those are generated by shadcn. If you need a variant, wrap the component.
- Page metadata goes through the Next.js metadata export. Use env.NEXT_PUBLIC_APP_URL for absolute URLs via absoluteUrl() in src/lib/utils.ts.

## Database rules

- We use Drizzle ORM with Postgres. The schema lives in src/db/schema/index.ts.
- Table ids are varchar(30) generated by generateId() from src/lib/id.ts with a type prefix, e.g. str_ for stores, prd_ for products. Never use auto increment integers.
- Money is stored as decimal(10,2) strings. Never store floats. Convert with Number() at the edge and keep cents math in integers.
- All tables need createdAt with defaultNow() and updatedAt.
- Run pnpm db:generate after schema changes to create a migration. Never edit generated SQL in drizzle/ by hand.
- Migration steps for production:
  1. Make schema change in src/db/schema.
  2. pnpm db:generate to create the migration file.
  3. Review the generated SQL in drizzle/.
  4. pnpm db:migrate against the staging database first. The connection string comes from DATABASE_URL in .env.
  5. Verify staging, then run against production with the prod DATABASE_URL.
  6. Commit the migration file in the same PR as the schema change.
- Never call db directly from client components. Data flows: client component -> server action (src/lib/actions) or RSC -> query function (src/lib/queries) -> db.
- Queries that return lists must be paginated. Look at src/lib/queries/product.ts for the pattern with limit/offset and total count.
- Soft deletes are not used. We hard delete and rely on Stripe/Clerk as the source of truth for payments and users.
- Indexes: add an index for any column used in a where clause on a hot path. Check existing indexes in the schema before adding.
- Seeding: pnpm db:seed runs src/db/seed.ts. It uses @faker-js/faker. Do not run it against production.
- The categories and subcategories are seeded from src/config/site.ts productConfig. If you add a category there, update the seed.

## Auth rules (Clerk)

- Auth is Clerk. Use auth() from @clerk/nextjs/server in server code and useUser() in client components.
- The middleware in src/middleware.ts protects /dashboard routes. Public routes are listed in the middleware config.
- User email: use getUserEmail() from src/lib/utils.ts, do not read emailAddresses[0] directly because the primary email is not always first.
- Roles are stored in Clerk private metadata. Stores have an ownerId column that maps to the Clerk user id.
- Never log Clerk user objects, they contain PII.

## Payments (Stripe)

- Stripe Connect is used for store payouts. Each store has a stripeAccountId.
- The platform fee is 1 percent, calculated in calculateOrderAmount in src/lib/checkout.ts. Amounts going to Stripe are integer cents.
- Webhooks land at /api/webhooks/stripe. Verify the signature with STRIPE_WEBHOOK_SECRET. Test locally with pnpm stripe:listen.
- Webhook events we handle: payment_intent.succeeded, payment_intent.payment_failed, account.updated, checkout.session.completed.
- Never trust amounts from the client. Recalculate the order total server side from the cart line items before creating a payment intent.
- Idempotency: order creation on payment_intent.succeeded must check for an existing order with the same paymentIntentId before inserting.

## API reference (internal)

POST /api/webhooks/stripe
  Headers: stripe-signature (required)
  Body: raw Stripe event payload, do not JSON.parse before signature verification
  Returns: 200 {"received": true} | 400 on bad signature

GET /api/og
  Query params: title (string, required), description (string, optional), type (string, optional)
  Returns: 1200x630 PNG open graph image rendered with @vercel/og
  Example: /api/og?title=Skateshop&description=An+open+source+storefront

POST /api/uploadthing
  Managed by uploadthing router in src/app/api/uploadthing/core.ts
  File rules: product images max 4MB, max 4 files, image/* only
  Auth: requires signed-in Clerk user, rejects otherwise

GET /api/products
  Does not exist. Product data flows through RSC queries, not REST. Do not add REST endpoints for data that RSCs can fetch.

## Testing rules

- Tests use Vitest. Test files are colocated: src/lib/foo.ts gets src/lib/foo.test.ts.
- Run pnpm test for a single run, pnpm test:watch during development.
- Pure functions in src/lib must have unit tests. UI components do not have tests yet, do not add testing-library without discussing first.
- Do not mock Drizzle in unit tests. Extract pure logic out of queries and test that. Query functions get covered later by integration tests (not set up yet).
- Test names describe behavior: "applies the platform fee to the order total", not "test calculateOrderAmount 2".
- Money assertions compare integer cents, never floats. expect(total).toBe(1099) not expect(total).toBeCloseTo(10.99).
- Run pnpm typecheck and pnpm lint before considering any task done. CI runs pnpm check which is lint + typecheck + format:check.
- Prettier config: no semicolons, double quotes, 2 space indent, import sorting via @ianvs/prettier-plugin-sort-imports. Run pnpm format:write if CI complains.

## Environment variables

Copy .env.example to .env. Required for dev:
- DATABASE_URL: Postgres connection string. Neon works, local Postgres works.
- NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY and CLERK_SECRET_KEY: from the Clerk dashboard, use a dev instance.
- STRIPE_API_KEY and NEXT_PUBLIC_STRIPE_PUBLISHABLE_KEY: Stripe test mode keys.
- STRIPE_WEBHOOK_SECRET: printed by pnpm stripe:listen when it starts.
- UPLOADTHING_SECRET and UPLOADTHING_APP_ID: from uploadthing dashboard.
- RESEND_API_KEY: only needed if you touch email, otherwise the newsletter form will error and that is fine in dev.
- NEXT_PUBLIC_APP_URL: http://localhost:3000 in dev.
Env validation lives in src/env.js with @t3-oss/env-nextjs. New env vars must be added to the schema there or the build fails. Set SKIP_ENV_VALIDATION=1 to build without env vars (Docker).

## Git workflow

- Commits follow conventional commits: feat:, fix:, chore:, docs:, refactor:, test:.
- Keep PRs under 400 lines of diff where possible. Split big changes.

## Code style misc

- No default exports except Next.js pages, layouts, and route handlers where the framework requires them. Everything else uses named exports.
- File names are kebab-case: add-to-cart-form.tsx, not AddToCartForm.tsx.
- Imports use the @/ alias for anything under src. No relative imports that climb more than one level (../../ is a smell).
- Prefer interface over type for object shapes, type for unions.
- Async server actions must return a result the client can act on, look at the ActionResult pattern in src/lib/actions/cart.ts.
- Error handling in server actions: wrap in try/catch and use getErrorMessage from src/lib/handle-error.ts. Never throw raw errors across the server action boundary.
- Zod schemas are named like the thing they validate plus Schema: cartLineItemSchema, updateProductRatingSchema.
- Do not use enums, use const objects with as const and derive the type.
- Avoid useEffect for data fetching. Data comes from RSCs or server actions.
- Comments explain why, not what. Delete commented out code, git remembers it.

## Deployment

- The app deploys on Vercel. Preview deployments are created for every PR automatically.
- Production deploys happen on merge to main. There is no manual deploy step.
- Environment variables in Vercel are managed in the dashboard, ask in #eng-infra if you need access.
- Build command is pnpm build which runs contentlayer build then next build. If contentlayer fails, check src/content mdx frontmatter first, it is usually a missing field.
- Bundle size: check the build output table. A page over 200kb first load JS needs a look. Use dynamic() imports for heavy client components like the image cropper.
- Cron: there are no cron jobs. If you think you need one, you probably want a Stripe webhook instead.

## Onboarding checklist (new contributors)

1. Install Node 20 LTS and pnpm 9 (corepack enable && corepack prepare pnpm@9.5.0 --activate).
2. Clone the repo and run pnpm install.
3. Copy .env.example to .env and fill in the values per the Environment variables section above.
4. Create a Neon database, paste DATABASE_URL.
5. Run pnpm db:push to sync the schema, then pnpm db:seed.
6. Create a Clerk dev application, enable email + Google sign in, paste the keys.
7. Create a Stripe test account, paste the test keys.
8. Run pnpm dev and open http://localhost:3000.
9. Sign up, create a store from /dashboard/stores, add a product, and buy it with card 4242 4242 4242 4242.
10. Run pnpm test and pnpm check to make sure your environment is healthy.

## Things that have bitten us before

- Stripe webhook handler must read the raw request body. Next.js route handlers parse JSON by default if you call req.json(), use req.text() and verify the signature on the raw string.
- Clerk middleware ordering matters. The matcher in src/middleware.ts must exclude /api/webhooks or Stripe calls get redirected to sign-in and fail silently.
- Drizzle decimal columns come back as strings. Number("12.99") is fine, parseFloat on undefined is not. Always handle the null case on optional decimal columns.
- contentlayer build is flaky on Node 22. Use Node 20.
- The product image cropper (react-cropper) breaks if rendered server side. It must stay behind a dynamic import with ssr false.
- uploadthing file URLs are permanent but the dev app and prod app have different app ids, so seeded image URLs from dev will 404 in prod. Seed prod separately.
- pnpm dlx shadcn-ui@latest add overwrites local changes in src/components/ui without asking. Diff before committing.
- Tailwind class sorting: the prettier plugin reorders classes, so a "no-op" format:write can produce a big diff on old files. Commit formatting separately from logic changes.
- date-fns format tokens: dd is day of month, DD is day of year. We have shipped that bug once already.
- Vercel preview deployments use the preview Clerk instance. Webhooks from Stripe test mode point at production URLs, so preview deployments never receive webhooks. Do not debug "missing orders" on previews.

## Common Drizzle snippets (copy these)

Get a store with its products:

```ts
const store = await db.query.stores.findFirst({
  where: eq(stores.id, storeId),
  with: {
    products: {
      limit: 10,
      orderBy: [desc(products.createdAt)],
    },
  },
})
```

Paginated product list with count (the standard pattern):

```ts
const [items, total] = await Promise.all([
  db
    .select()
    .from(products)
    .where(and(...conditions))
    .limit(limit)
    .offset(offset)
    .orderBy(desc(products.createdAt)),
  db
    .select({ count: count() })
    .from(products)
    .where(and(...conditions))
    .then((res) => res[0]?.count ?? 0),
])
```

Insert with a generated id:

```ts
await db.insert(stores).values({
  id: generateId("store"),
  name: input.name,
  ownerId: user.id,
})
```

Transaction (use for multi-table writes like order + order items):

```ts
await db.transaction(async (tx) => {
  await tx.insert(orders).values(order)
  await tx.insert(orderItems).values(items)
})
```

## Stripe webhook payload notes

payment_intent.succeeded, the fields we actually read:

```json
{
  "id": "evt_...",
  "type": "payment_intent.succeeded",
  "data": {
    "object": {
      "id": "pi_...",
      "amount": 12999,
      "currency": "usd",
      "metadata": {
        "cartId": "crt_...",
        "storeId": "str_..."
      }
    }
  }
}
```

The cartId and storeId in metadata are set by us when creating the payment intent in src/lib/actions/stripe.ts. If metadata is missing the webhook handler logs and returns 200 anyway, returning 4xx makes Stripe retry for days.

account.updated: we only care about charges_enabled and details_submitted flipping to true, that is when a store becomes able to sell. The handler updates the store's stripeAccountId status.

## Tailwind tokens

- Colors come from CSS variables defined in src/styles/globals.css, mapped in tailwind.config.ts. Use semantic names: bg-background, text-foreground, text-muted-foreground, border-border, bg-primary, bg-destructive.
- Never hardcode hex values in className. If the design needs a new color, add a CSS variable for both light and dark in globals.css first.
- Radius: rounded-lg uses var(--radius), rounded-md is calc(var(--radius) - 2px). Keep cards at rounded-lg.
- Container is centered with 2rem padding and 1400px max width, configured in tailwind.config.ts.
- Animation utilities come from tailwindcss-animate. Accordion animations are pre-wired (accordion-down, accordion-up).

## FAQ

Q: pnpm dev fails with "Invalid environment variables".
A: Your .env is missing something. Read the error, it names the variable. See the Environment variables section.

Q: Why does the build run contentlayer first?
A: The blog and docs pages import generated types from .contentlayer. Without it next build fails on missing imports.

Q: Can I use fetch to call our own API routes from a server component?
A: No. Call the query function directly. Fetching your own origin in RSC wastes a round trip and breaks on preview URLs.

Q: Where do I put a new server action?
A: src/lib/actions/<domain>.ts. Add "use server" at the top of the file, not per function. Validate input with the matching zod schema from src/lib/validations.

Q: Why is my decimal price showing as a string?
A: Drizzle returns decimal columns as strings by design. Format at the edge with formatPrice from src/lib/utils.ts.

Q: How do I add a new product category?
A: Update productConfig in src/config/site.ts, update the seed in src/db/seed.ts, regenerate the db, and check the category icons mapping in src/components/icons.ts.
